# LinguaCafe AI 协作与交付规则

> 当前权威版本：2026-07-18。
> 根级硬规则见 `AGENTS.md`；本文件解释任务如何落位、执行和验收。业务状态见 `docs/DOCUMENTATION_INDEX.md`。Git 历史保留旧版长文，不在当前上下文继续携带已停用工作流。

## 0. 权威、目标与阅读方式

1. 本文件不得覆盖 `AGENTS.md`、已接受 ADR 或用户当前明确决定。
2. 新任务先从文档索引进入，只读当前模块需要的章节；不把本文件、master plan、全部 ADR 和字幕一次性塞进上下文。
3. 本文件只保留长期协作流程。GLM/OpenCode/CodeBuddy/WorkBuddy 等具体工具或旧角色不是架构事实，不作为当前硬规则。
4. 规则的目标是降低未来决策次数和越界概率，不是让提示词更长、任务更复杂或模型工作更久。

## 1. 决策责任

- 用户决定产品目标、优先级、风险授权和是否接受长期规则。
- 执行 Agent 负责读取真实代码、提出最小方案、实施、运行外部验证并报告事实。
- AI 可以起草 spec、ADR、测试和架构候选，但不得把自己的建议自动升格为产品决定或长期规则。
- 报告、测试绿灯和 Agent 自审都不能代替用户的产品判断。

### 1.5 当前执行模型（兼容锚点）

当前任务由一个主执行 Agent 对结果负责；是否使用内部工具或子代理不改变范围、审批、验证和停止规则。旧 CodeBuddy / WorkBuddy / GLM 接力流程仅是历史，不再生效。

## 2. 任务进入条件

每个任务在实施前冻结一个小切片：

1. 目标与不做事项。
2. 当前事实和根因证据。
3. 模块责任、接口/seam、数据流。
4. 允许文件与禁止文件。
5. 受保护行为和数据写入边界。
6. 最小可执行验收与停止条件。

纯文档、纯验证、单文件修复可以独立成任务。不得为了“复合任务”、复杂度分数或进度百分比强塞架构改造、功能开发、DTO、Service 或额外测试。

## 3. 上下文与文档分层

### 3.1 五层上下文

1. `AGENTS.md`：每次加载的稳定硬规则。
2. `docs/DOCUMENTATION_INDEX.md`：只负责路由当前权威文档。
3. ADR / 模块契约：按受影响模块加载。
4. 源码、接口和相关测试：按任务加载。
5. 错误输出、临时计划和对话历史：仅本轮使用。

历史状态、已关闭报告、一次性调试记录不得混入当前指令。文档冲突时先查状态和权威来源；无法消解就停止，不让模型猜。

### 3.2 文档职责

| 信息 | 唯一落点 |
|---|---|
| 长期安全、范围、停止规则 | `AGENTS.md` |
| 当前入口和文档路由 | `docs/DOCUMENTATION_INDEX.md` |
| 当前工作台 | `docs/plans/current-working-handoff.md` |
| 长期工作登记 | `docs/plans/linguacafe-master-plan.md` |
| 稳定技术决定及理由 | `docs/adr/` |
| 模块责任、接口与数据流 | `docs/architecture/` 或模块 contract |
| 可执行不变量 | tests / smoke / harness |
| 旧状态和执行报告 | `docs/history/` |

同一规则只能有一个权威正文；其他文档只链接，不复制。未来计划不得写成已实现，临时修复不得写成永久原则。

## 4. Spec 生命周期与 Harness

### 4.1 规则晋升门槛

候选决定必须由用户明确冻结；除核心产品章程/排除项外，还要经过真实实现/验收，再选择落点。只有稳定、重复相关、违反代价高且可验证的决定才进入 spec、ADR 或硬规则。

每次纠正 AI 时问：“这会再次发生吗？”

- 否：只修当前问题。
- 会，但决定仍在变化：放 task plan。
- 会，决定已稳定：写 ADR / 模块 contract。
- 会，且破坏后难发现或代价高：同时加最小 test / smoke / harness。

### 4.2 Harness 最低标准

Harness 必须能独立完成：准备条件、执行真实路径、输出明确通过/失败。优先保护：

- 已稳定的核心用户流程。
- 权限、数据保存/删除、导入导出、ReviewLog/FSRS 等高风险写入。
- 重复出现的回归。
- 一旦出错很难定位的跨模块链路。

不追求覆盖率数字，不为每条自然语言 spec 配测试。一个公开入口上的承重检查优于多个调用方的重复 guard。

## 5. Architecture Gate

风险等级与审批语义以 `docs/adr/ADR-0001-architecture-gate-workflow.md` 为准。

- 低风险：单文件、纯样式、纯文档、明确根因的小修。限制范围并做相称验证。
- 中风险：不改变既有语义的 props/events、前端 API 调用、工具函数或多文件同模块变更。按 ADR-0001 做最小上下文、必要的接口设计和实施后质量审查。
- 高风险：Vuex/store 逻辑、后端 endpoint、请求/响应 payload 或公开接口语义、跨模块、数据模型、核心写入、FSRS、WordSense、ReviewCard、AI、reader 状态流、import/export。先做任务范围架构审查，用户确认后实施。

架构审查必须回答：谁拥有责任、接口是什么、复杂度藏在哪里、会跨哪些 seam、允许/禁止文件、兼容风险、验证方式、是否需要 ADR。

高内聚低耦合不等于拆得越碎越好。删除一个模块若只会把同样复杂度搬到调用方，该模块没有带来 depth；只有真实的第二个调用者或适配器出现时，才考虑新增抽象 seam。

## 6. 实施纪律

1. 先读将修改的完整文件、全部调用方、相关接口和测试。
2. 优先修共享根因，避免在每个调用方打补丁。
3. 复用 upstream、现有 helper、框架能力和已安装依赖。
4. 不顺手清理，不提前搭“以后可能用”的脚手架。
5. 不并行修改同一文件，不并行运行争用同一 testing 数据库的写测试。
6. 每个小切片保持可运行；发现范围扩大或新产品决定立即停止。
7. 架构变化与功能变化能分开验证时分开提交；纯机械整理不得掩盖行为变化。

## 7. 验收闭环

验收按风险选择，不机械执行全仓测试：

1. 静态范围检查：最终 diff、允许文件、禁止模式、秘密和生成物。
2. 聚焦自动测试：覆盖修改接口、正常路径、拒绝路径和关键副作用。
3. 受保护模块回归：按 `AGENTS.md` 验证矩阵触发。
4. 前端构建：前端或前后端契约变化时运行。
5. 真实页面验收：可见 UI、reader、review、import/export 变化时运行，并检查 Console / Network。

API、源码阅读、截图或 Agent 自述不能代替真实页面操作。环境无法提供要求的验收时报告 `Incomplete`，不得伪造或偷换验证方式。

## 8. 安全红线

`AGENTS.md` 是安全红线的唯一权威正文，本文件不复制清库、凭据、权限、写入、Git、任务外文件或完成 hook 规则。任务提示词只能收紧这些边界，不能放宽。

### 8.7 Anki 参考优先（兼容锚点）

涉及 Anki 评分、队列、自定义学习、卡面、字段、快捷键、撤销、学习步骤或复习顺序时，决策顺序固定为：Anki 官方手册/源码 → 保留核心语义的 LinguaCafe Web 适配 → Anki 没有对应设计时再做独立产品判断。不得用普通网页习惯覆盖 Anki 已有语义；Anki 未冻结视觉样式时，ADR 或计划必须明确标注“LinguaCafe 项目适配”。论坛和第三方模板不能冒充官方默认。

## 9. 报告与 Git

最终报告必须能由 Git 和命令输出复验：

- 实际改动文件及原因。
- 是否越界、是否触及数据或接口。
- 实际运行的命令、退出结果和未验证项。
- 浏览器结果或 `Incomplete`。
- branch、commit、push 状态。

只暂存精确文件。报告禁止写“应该通过”“大概完成”或虚构进度百分比。

## 19. 任务规模规则（取代复杂度分数）

任务规模由风险和可验证 seam 决定，不设最低复杂度、运行时长或百分比。

- 一个任务只交付一个可陈述结果。
- 文件数或 seam 明显增长时拆分。
- 小修不与无关架构工作捆绑。
- 高风险工作宁可缩小垂直切片，也不降低验证。

## 27. 模块边界与专项验证

### 27.0 第一原则（兼容锚点）

代码安全和稳定性优先于功能速度；具体安全、模块和写入不变量只以 `AGENTS.md` 为权威。

### 27.1 长期模块不变量

稳定模块责任、正式评分入口和 AI/只读边界只以 `AGENTS.md` §6 为权威。模块契约和 ADR 解释实现细节；本文件不复制类名、payload 或禁令。

### 27.5 页面与词元 smoke

涉及真实 token 点击时，按 `docs/plans/mcp-chrome-local-smoke-playbook.md` 和 `docs/plans/morphology-test-sample-tracker.md` 选新样本并真实点击。API/fetch/axios 不能冒充页面操作；工具不可用则标记 `Incomplete`。

### 27.7 Testing DB 健康检查

涉及 Feature tests 或数据库写测试时，先运行：

```bash
php artisan test --filter=TestingDatabaseHealthTest
php artisan test --filter=TestingDatabaseHealthConfigTest
```

失败时按 `docs/plans/testing-db-health-playbook.md` 处理；数据库安全边界见 `AGENTS.md` §9。

### 27.8 PHP 测试输出

高输出命令写入已忽略的临时日志，只返回退出码和末尾摘要。Unit 可独立运行；Feature 按文件或业务模块分组。失败处理的硬规则见 `AGENTS.md` §8。

## 28. 需求放置与报告闭环

### 28.1 单一权威来源

本节是需求落位与报告闭环的唯一正文；其他文档只链接和记录本轮结果。

### 28.2 需求处理

新需求先分类为：当前阻塞、当前兼容、未来阶段、独立主线或暂不进入计划。只有当前阻塞或天然兼容的需求可进入当前切片；其余登记到正确 plan/roadmap，不自动抢占下一任务。

落位必须说明阶段、理由、前置条件和为什么不提前做。位置不明确时冻结并询问用户，不让执行 Agent 决定产品优先级。

### 28.3 架构检查与开发的关系

所有代码任务都要做与风险相称的架构检查，但不强制每个任务同时产生架构改造和功能代码。没有真实结构问题时，记录“无需结构改动”即可；禁止为了流程形式制造抽象、拆分或文档。

### 28.4 字幕使用

字幕是经验来源，不是项目权威。只按任务读取相关字幕；不得让字幕覆盖代码事实、ADR 或用户最新决定。采用字幕建议前，必须将其转为本项目可验证的触发条件；探索性意见不进入 spec。

本轮长期规则整合参考了用户提供的 9 份字幕，采用的稳定原则为：规则按需加载、spec 滞后于稳定决定、反向边界优先、关键不变量进入 harness、AI 延续而不创造架构、模块化需平衡接口成本、验收发生在 AI 输出之外。

### 28.5 报告处理

验收依次核查：用户要求 → 最终 diff → 允许范围 → 测试/构建/浏览器证据 → 当前计划和未决产品选择。Accept、Refuse、阶段性 Accept 或 Incomplete 必须基于证据，不只依据 Agent 报告。

### 28.6 下一任务

完成当前任务后停止。只有用户明确要求继续且下一切片边界已确定时，才进入下一任务；不得把建议、future work 或报告中的 follow-up 自动实施。

## 29. 规则收敛检查

修改长期规则时最多进行三轮独立对抗审查。每轮按以下顺序分类：权威冲突、行为/安全漏洞、范围膨胀、无法验证、纯措辞。前四类必须修正；只剩纯措辞或同义建议时停止。规则越改越长本身不是进步，能删除重复且不丢失约束才算收敛。
