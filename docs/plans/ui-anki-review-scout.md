## 2. 当前 SenseReview 结构

### 2.1 顶部统计栏（8 chips + 1 alert）

```
到期数量 | 已复习 | 剩余 | 今日已复习 | 今日重置 | 总词义卡 | 启用中 | 已归档 | 当前到期
```

其中：
- `到期数量` 和 `当前到期` 含义高度重叠
- `今日重置`、`总词义卡`、`已归档` 是管理信息，非复习必需
- `启用中` 是管理信息

### 2.2 卡片主区（全部展开）

```
行 1: lemma | surface_form / pos           | [fsrs_state] [X 次]
行 2: [编辑] [查看原文] [归档] [重置] [彻底删除]  ← 5 个按钮
行 3: 中文释义 | 例句(英/中)
      英文释义 | FSRS 表格(到期/稳定度/难度/遗忘次数)
      近义译法 |
      搭配     |
行 4: [忘了] [勉强记得] [记得] [很熟]  ← 评分按钮
```

### 2.3 问题识别

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 无"显示答案"流程 | 🔴 高 | 问题和答案同时展示，用户跳过了回忆过程 |
| 5 个管理按钮全部展开 | 🔴 高 | 编辑/归档/重置/删除占据视觉焦点 |
| FSRS 数值全部可见 | 🟡 中 | 90% 用户不需要每天看到稳定度和难度 |
| 8 个 stats chips 过于拥挤 | 🟡 中 | 含大量管理信息 |
| 评分按钮不显示下次间隔 | 🟡 中 | 用户不知道选每个按钮的结果 |
| 无键盘快捷键 | 🟡 中 | SenseReview 无 keyup 监听（Review.vue 已有） |
| 例句/搭配/近义译法全部展开 | 🟢 低 | 这些属于语境信息，可折叠但非阻塞 |

## 3. Anki 刷卡节奏原则

### 3.1 问题先行
用户先看到问题（正面），思考答案后再展示答案（背面）。这个"回忆"过程是 spaced repetition 的核心学习机制。

### 3.2 答案后显
显示答案后，用户对比自己的回忆与正确答案，然后评分。评分按钮同时展示下次间隔。

### 3.3 评分即调度
每个评分按钮直接触发 FSRS 调度计算。用户选择评分后自动进入下一张卡，不打断节奏。

### 3.4 间隔可见
Anki 在每个评分按钮旁显示选择该评分后的下次间隔（Again: 1m, Hard: 10m, Good: 1d, Easy: 4d）。用户能预见评分后果。

### 3.5 低频操作收纳
编辑、标记、暂停、置底等操作放在"More"或"Edit"下拉菜单。复习区不出现管理操作。

### 3.6 快捷键降低摩擦
Space 显示答案，1/2/3/4 快速评分。用户可不碰鼠标完成整个复习流程。

### 3.7 复习页不当管理页
信息层级：复习 > 编辑 > 管理。不要在用户需要"回忆词义"的时刻展示"彻底删除"按钮。

## 5. 推荐目标体验

### 混合型方案（推荐，待用户确认）

#### 阶段 1：默认极简（打开页面时）

```
┌──────────────────────────────────────┐
│  lemma / surface_form (pos)          │
│                                      │
│  "这个句子里的 X 是什么意思？"        │
│  或是直接显示例句中高亮目标词          │
│                                      │
│  ┌──────────────────────────────┐   │
│  │  显示答案 / Show Answer       │   │
│  └──────────────────────────────┘   │
│                                      │
│  提示：Space 显示答案，1/2/3/4 评分   │
└──────────────────────────────────────┘
```

#### 阶段 2：显示答案后

```
┌──────────────────────────────────────┐
│  lemma / surface_form (pos)          │
│                                      │
│  词义: sense_zh / sense_en           │
│  例句: 英文 + 中文译文                │
│                                      │
│  [查看原文]  [► 更多信息]             │
│                                      │
│  [忘了(1m)] [勉强(10m)] [记得(+1d)] [很熟(+4d)]  │
│                                      │
│  提示：1 忘了 / 2 勉强 / 3 记得 / 4 很熟 │
└──────────────────────────────────────┘
```

#### 更多信息展开（点击"► 更多信息"后）

```
┌──────────────────────────────────────┐
│  近义译法：...                        │
│  搭配：...                           │
│  FSRS 详情（折叠）：                  │
│    ─ 到期 2026-06-28                  │
│    ─ 稳定度 12.3                      │
│    ─ 难度 3.8                         │
│  [编辑] [归档] [重置] [彻底删除]       │
└──────────────────────────────────────┘
```

### 关键设计决策

1. **问题区域显示什么？** 建议显示 lemma + 例句（目标词高亮）+ "这个词在这里是什么意思？"
2. **答案区域显示什么？** sense_zh + sense_en + 例句译文（确认核心理解）
3. **More 里放什么？** 编辑 / 查看原文 / 归档 / 重置 / 彻底删除 / FSRS 详情
4. **顶部 stats 精简为：** 到期 / 已复习 / 剩余 / Today（今日已复习）
5. **间隔显示：** 先做语义提示（很快再来 / 稍后复习 / 按计划 / 更久以后），再做精确间隔。

## 6. 分阶段实现建议

### UI-Review-a：纯前端信息层级整理（推荐第 1 步）

**范围**：仅改 SenseReview.vue template + data

**不改后端**：不改调度、不改 API、不改评分

**具体内容**：
1. 收起 5 个管理按钮到 More 下拉菜单（`v-menu` 或 `v-btn-toggle`）
   - More 中包含：编辑、查看原文、归档、重置、彻底删除
   - 主界面只留评分按钮 + 显示答案按钮
2. FSRS 数值表格折叠到一个可展开区域
   - 默认：只显示"到期时间"在一行
   - 点击展开：显示完整稳定度/难度/遗忘次数
3. 顶部 stats 精简
   - 默认：到期数量 / 已复习 / 剩余（第一行）
   - 其他统计放入"详情"折叠或直接移除
4. 近义译法和搭配移到"更多信息"区域
   - 主区域只保留 lemma + sense_zh/sense_en + 例句

**风险**：低（纯前端改动，不影响任何后端行为）
**影响**：直接降低认知负担，用户不再被 5 个管理按钮分散注意力

### UI-Review-b：Anki-like 显示答案流程（推荐第 2 步）

**范围**：SenseReview.vue template + computed + methods

**不改后端**：不改调度、不改 API

**具体内容**：
1. 增加 `showAnswer` 状态变量（boolean）
2. 初始：只显示问题（lemma + 例句高亮目标词 + "点击显示答案"按钮）
3. 点击"显示答案"或按 Space 后：显示答案区 + 评分按钮
4. 答案区包括：sense_zh / sense_en / 例句译文
5. 评分后：自动加载下一张卡，重置 showAnswer = false

**风险**：中（需要仔细管理状态流转、确保不引入 bug）
**注意**：不要改变 `rate()` API 的调用方式

### UI-Review-c：键盘快捷键（推荐第 3 步）

**范围**：SenseReview.vue template + mounted + keyup handler

**不改后端**

**参考**：Review.vue 已有 Enter/1/2/3/4 快捷键实现，可复用模式

**具体内容**：
1. 监听 `keyup` 事件
2. Space → 显示答案（或进入下一张卡）
3. 1 → 忘了（again）
4. 2 → 勉强记得（hard）
5. 3 → 记得（good）
6. 4 → 很熟（easy）
7. 复用或参考 `ReviewHotkeyInformationDialog.vue` 提示弹窗

**风险**：低（纯前端，不影响 API）
**注意**：确保输入框（如搜索框）获得焦点时不触发快捷键

### UI-Review-d：评分按钮显示下次间隔侦察/设计（推荐第 4 步）

**范围**：可能需要后端微调

**评估**：
1. 当前 `POST /reviews/senses/{id}/rate` 返回 summary 和更新数据，但不返回每次评分对应的间隔
2. 如果要在前端显示下次间隔，有两种方案：

**方案 A：语义提示（推荐先做）**
- 不依赖后端计算
- 在按钮上加文字："很快再来"、"稍后复习"、"按计划复习"、"更久以后"
- 纯前端文案，不改变 API

**方案 B：精确间隔（需要后端新增能力）**
- 需要后端在评分前预估每个 rating 的下次间隔
- 需要新增类似 `preview-rating` 端点
- 需要修改 FsrsSchedulingService 暴露预估方法
- 建议**独立成后续任务**，不做为 UI-Review 系列的一部分

### UI-Review-e：真实浏览器 smoke + 人工体验验收（第 5 步）

**范围**：验收

**内容**：
1. 用户实际刷 5-10 张卡
2. 检查节奏是否顺手
3. 检查按钮布局是否误导
4. 检查移动端 / 窄屏
5. 检查 Console 错误
6. 如果 Playwright 登录已修复或人工登录完成

## 7. `/review` legacy 页面评估

### 现状
- `/review`（Review.vue）仍存在，承载旧的 word/phrase/sense 混合复习逻辑
- 它比 SenseReview 更成熟：**已有显示答案流程**（Enter/Space），**已有快捷键**（1/2/3/4），**已有 More 菜单**
- 但它是混合逻辑，与 sense-only 主线冲突

### 判断
1. **今晚不要改路由**：不要删除 `/review`，不要重定向
2. **标记为 legacy**：在 roadmap 上标注 `/review` 不再积极开发
3. **长期方向**：引导用户使用 `/reviews/senses`，并逐步将 Review.vue 的优秀特性（快捷键、显示答案流程）移植到 SenseReview.vue
4. **Review.vue 仍可保留** 给有 legacy word 数据的用户，但新用户应直接使用 `/reviews/senses`
