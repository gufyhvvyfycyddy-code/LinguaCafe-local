# AI 阅读辅助计划

> 本文档描述 LinguaCafe 的 AI 阅读辅助功能产品设计和分阶段开发计划。
>
> **性质**：纯文档规划，不进入本轮实现。
> **状态**：设计草案 + schema 实验完成（待真实模型调用验证）。
> **相关实验文档**：[ai-reading-assist-schema-experiment.md](./ai-reading-assist-schema-experiment.md)

---

## 1. 总目标

LinguaCafe 的 AI 阅读辅助不是替代词典，而是补足词典在语境词义、词组、固定搭配、句子译文上的不足。

系统提供两种入口：

1. **手动模式（优先）**：用户复制整章英文 + 固定提示词，发送给 DeepSeek / GPT，再粘贴返回内容到系统。
2. **自动模式（后续）**：配置 API 后由系统直接调用 AI 分析本章。

两种入口最终都返回系统可识别的结构化文本，经过预览验证后写入学习数据。

## 2. AI 译文按句显示

### 2.1 显示方式

- 阅读页增加"显示 AI 译文"切换按钮。
- 点击后，每句英文句子下方出现一行中文译文。
- 再次点击后隐藏译文，恢复纯英文阅读。
- 译文按句一一对应，不是一整块堆在文章底部。

### 2.2 布局限制

- 中文译文通常比英文句子短。
- 如果中文译文容纳不下，优先调整英文句子的行距/词间距/句子块布局，让英文下方能容纳中文。
- 第一版不做左右并排，优先上下句对齐。

### 2.3 不影响已有数据

- AI 译文是阅读辅助层，不改变原文。
- 不影响词频统计。
- 不影响复习调度。
- 不自动创建复习卡。

---

## 3. AI 生词/词组释义

### 3.1 AI 只给一种释义

- AI 只输出本语境中的一种释义（中文或英文）。
- 不要求列所有词典义项。
- AI 释义不自动创建复习卡。

### 3.2 用户决定是否添加

- 用户点击词或词组后，才能选择是否将 AI 释义添加为 WordSense。
- AI 识别出的词组不自动进入复习。

### 3.3 词组的处理

- 当用户点击词组中的某个词时，系统可提示"这个位置可能是固定搭配"。
- 用户可选择：
  - 添加整个词组到复习。
  - 只添加当前单词到复习。
  - 不添加。
- 如果用户只添加当前单词，而 AI 只返回了词组释义，系统不强行生成单词释义，由用户自己根据译文判断。

## 4. 原型选择与手动原型

### 4.1 添加释义时必须选择原型

- 添加 AI 释义或词典释义时，用户必须能选择添加到哪个词形。
- 可选项包括：
  - **当前词形**：原文中的形态，例如 `investigates`
  - **系统推测原型**：例如 `investigate`（来自 tokenizer 的 lemma 字段）
  - **用户手动输入原型**：不受系统推测限制

### 4.2 手动原型的范围

- 用户手动输入的原型只作用于本篇文章当前词的绑定。
- 不应全局改写所有 encountered_words 的 lemma。
- 例子：原文是 `investigates`，用户可以把释义添加到 `investigate`，这张释义卡仍然绑定当前包含 `investigates` 的来源句。

### 4.3 代词等特殊情况

- `theirs` / `their` / `they` 这类代词原型关系不能强行由 AI 或系统自动决定。
- 用户必须可手动选择或输入原型。

---

## 5. 释义卡片与多例句绑定

### 5.1 释义卡可绑定多例句

- 一张释义卡可以绑定多个来源例句。
- 用户查词时，必须先确定绑定哪张释义卡。

### 5.2 例句的展示顺序

- 只有用户选定释义卡后，系统才展示"这张释义卡已绑定的其他例句"。
- 展示其他例句的目的：
  - 帮用户判断当前句子是否属于同一个意思。
  - 帮用户决定绑定旧卡还是新建释义卡。
- 不应在用户还没选定释义前就展示其他例句，因为系统不知道用户要看哪个意思。

---

## 6. 复习卡片例句轮换

### 6.1 例句轮换目标

- 复习释义卡时，例句来自这张卡绑定过的多个来源句。
- 不应永远显示第一条例句。
- 需要尽量平均轮换。
- 避免连续两天出现同一句。

### 6.2 第一版实现

- 第一版可以先实现简单随机轮换。
- 但必须保留"平均轮换"的后续目标。
- 后续可记录每个例句的展示历史，用于更公平地轮换。

---

## 7. 查词侧栏 UI 简化

### 7.1 保留的项目

1. 单词原型追溯。
2. 标记为已知。
3. 忽略单词。
4. 恢复新词。
5. 删除词条（后续评估改为归档/隐藏）。
6. 词典结果。
7. AI 释义。
8. 选择释义后展示该释义卡绑定的其他例句。

### 7.2 移除或默认隐藏的项目

1. 1-7 熟练度条。
2. 对号/叉号状态按钮。
3. 大量空词性分类。
4. 操作记录。
5. 常驻"关于本章"。
6. 常驻阅读设置。
7. 默认展开的词典结果。

### 7.3 文案调整

- "旧词条释义" → "选择释义"
- "添加新释义"简化为只显示：
  - 词性
  - 中文释义
- 英文释义、例句、近义用法、搭配放入高级编辑，不默认展示。

---

## 8. Hover 自动查词开关

- 鼠标放到单词上自动查词应变成可开关功能。
- 默认是否开启待用户确认。
- 如果用户没有点击查词，可以默认认为用户认识或暂时不处理该词。
- 该功能应放入阅读设置。
- 阅读设置本身不应常驻占用右栏，而应通过按钮打开。

---

## 9. AI 输出稳定性

### 9.1 第一阶段不做实时 API 调用

- 第一版做"复制整章英文 → 用户发 AI → 用户粘贴返回 → 系统解析"流程。
- 先通过手动流程验证 AI 输出格式稳定性。

### 9.2 严格 JSON schema

- 设计严格 JSON schema，让 AI 只返回可解析 JSON。
- prompt 中明确要求只返回 JSON，不返回多余内容。

### 9.3 Schema 实验

- 实验 DeepSeek Flash 和 DeepSeek Pro 的返回稳定性。
- 实验不同 prompt 对各模型的准确性。
- 详细实验材料、schema 设计、提示词模板、对比表见 [ai-reading-assist-schema-experiment.md](./ai-reading-assist-schema-experiment.md)。
- **用户明确要求比较两个模型**：DeepSeek Flash（低成本快速）和 DeepSeek Pro（高精度）。
- **用户不希望手动修 AI 输出**，要求尽量做到粘贴后直接可用。

### 9.4 导入时容错

- 系统导入时支持：
  - 识别 JSON 代码块（```json ... ```）。
  - 去掉多余 Markdown。
  - 给出清晰错误提示。
  - 不要求用户手动改格式。
- 如果无法完全稳定，必须先做"导入前预览 + 错误定位"，不要直接写入学习数据。

## 10. AI API 配置页（后续阶段）

### 10.1 配置项

- 默认可选供应商：
  - DeepSeek Flash
  - DeepSeek Pro
  - OpenAI-compatible 自定义

### 10.2 用户填写内容

- API key（必填）
- 供应商（必选）
- 模型（必选）
- base URL（高级设置，可选）

### 10.3 安全存储要求

- API key 不能明文普通存储。
- 不允许写入 `.env`。
- 优先使用安全加密存储或本地用户设置方案。

### 10.4 第一阶段不做 API 配置

- 第一版先做手动复制 prompt，不急着存 API key 或做 API 调用。

## 12. 用户产品决定（2026-06-29）

以下为网页端 GPT 确认的产品决定，后续开发必须遵守：

1. **模型对比**：Schema 实验必须比较 DeepSeek Flash 和 DeepSeek Pro 两个模型。
2. **输出稳定性**：用户不希望手动修 AI 输出，要求尽量做到粘贴后直接可用。
3. **原型选择**：添加释义时必须支持当前词形、系统推测原型、手动输入原型三种方式。
4. **手动原型范围**：手动输入的原型只作用于本篇当前词的绑定，不全局修改 lemma。
5. **Hover 开关**：自动查词必须变为可开关功能。
6. **例句展示时机**：释义卡绑定的其他例句只在用户选定释义后展示。
7. **例句轮换**：复习卡例句轮换要尽量平均，避免连续两天出现同一句。
8. **译文位置**：AI 译文按句显示在英文句子下方，上下对齐，不做左右并排。
9. **侧栏瘦身**：查词侧栏移除熟练度条、对号按钮、操作记录、常驻"关于本章"、常驻阅读设置。

## 13. 本轮不做

1. 不自动批量创建复习卡。
2. 不直接信任 AI 释义（必须先预览）。
3. 不自动把词组加入复习。
4. 不强行全局修改 lemma。
5. 不把 API key 写入 `.env`。
6. 不在本轮实现代码。
