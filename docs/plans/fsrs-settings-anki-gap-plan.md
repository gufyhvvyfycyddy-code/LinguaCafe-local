# FSRS 设置页 Anki 对标与拆解计划

> **创建日期**：2026-06-25
> **关联里程碑**：[linguacafe-fsrs-roadmap.md](./linguacafe-fsrs-roadmap.md) → D 系列

---

## 1. 当前 LinguaCafe 已有能力

LinguaCafe 的 FSRS 设置页位于管理后台 (`/admin` → 间隔重复系统)，是一个全局设置页，以 `AdminReviewSettings.vue` 组件渲染。

**✅ 已有且真实可用的功能：**

- **Desired Retention 下拉选择**：提供 8 个选项（70%, 75%, 80%, 85%, 90%, 92%, 95%, 97%），默认 90%。通过 `v-select` 下拉调整，调用 `POST /settings/global/update` 保存 `fsrsDesiredRetention` 到数据库。保存成功后有绿色提示，说明新评分会使用该保持率但已排程卡片不会自动重排。

- **保存按钮**：页面底部有蓝色"保存 FSRS 设置"按钮，带 loading 状态和成功/失败提示。

- **FSRS 说明卡片**：页面顶部有蓝色信息卡片，用通俗中文解释 FSRS 是什么：系统根据评分、稳定度、难度和复习历史自动计算下次复习时间。明确说明"不建议也不允许手动编辑参数"，"如需重新开始，应使用重置卡片为新学状态"。

- **FSRS 统计总览**：另一张卡片展示当前语言的词义复习卡统计，按语言区分：

  - 概况（总词义卡 / 启用中 / 已归档 / 当前到期）
  - 状态分布（新卡 / 学习中 / 复习中 / 重新学习）
  - FSRS 熟练度（平均稳定度 / 平均难度 / 总遗忘次数 / 今日已复习 / 今日重置）

  统计数据来自 `GET /review-cards/stats` 端点（`ReviewStatsService` 实时计算）。

- **旧 SRS 设置折叠区**：可展开/折叠的旧版间隔复习间隔设置，仅影响 legacy word card 和 phrase，另有警告说明词义复习卡不受此影响。

- **卡片重置**（⚠️ 不在本页）：单卡/批量重置功能在 `/review-cards/manage` 管理页实现（操作列"更多"菜单 → 重置），不在 FSRS 设置页本身。

**⚠️ 当前是占位文字的功能：**

- **参数来源**：显示"fsrs-rs-php 默认参数"，是固定文案，不反映真实参数来源、版本号或优化历史。
- **参数编辑**：显示"暂未开放"，完全不可交互。
- **卡片重置**：显示"后续开放"，实际上重置功能在管理页面已经做了，但这页的说明没有更新，也没有跳转入口。
- **参数优化**：显示"后续开放"，无任何交互入口。
- 页面底部的注释："FSRS 参数优化和卡片重置功能会在后续版本加入。" —— 这个说明已过时，因为卡片重置在管理页已实现。

## 2. 与 Anki FSRS 设置的差异

| 能力 | Anki | LinguaCafe 当前 | 建议 |
|------|------|----------------|------|
| **Desired Retention** | 滑块 (0.70-0.99)，默认 0.90，实时显示复习负担估算 | 下拉 (70%-97%), 默认 90%，有"越高复习负担越重"说明，无负担估算 | 增加每个 retention 值的负担预估，帮用户理解后果 |
| **Help Me Decide** | 用模拟结果帮用户选 retention，回答"我每天大概复习多少" | ❌ 无此功能 | D.2-c 可做简易版：按当前卡片数 × retention 对应复习频率做粗略预估 |
| **参数优化** | 根据复习历史自动生成更适合用户的 FSRS 参数，需积累一定复习数据 | ⚠️ 占位文字"后续开放"，无按钮 | D.3-a 可实现：调用 fsrs-rs-php 的 optimize 方法，优化前检查 ReviewLog 数量是否足够 |
| **参数健康检查** | 检查复习历史是否足够优化，数据不足时不建议优化 | ❌ 无此功能 | D.3-a 加入前置检查：ReviewLog < N 条时显示警告，不允许执行优化 |
| **重排已有卡片** | 改 retention 或参数后可选择重排，默认不自动重排，强提醒会产生大量到期卡 | ❌ 无此功能，retention 保存后明确说"不自动重排" | D.4-a 可做手动触发：显示影响卡片数量，用户强确认后才执行 |
| **模拟未来复习负担** | 展示"每天大概要复习多少张 / 花多少时间" | ❌ 无此功能 | 优先级较低，D.2-c 做简易版后可在后续版本升级为图表 |
| **参数来源 / 版本** | 显示当前参数来源（默认/优化）、优化时间 | 固定文字"fsrs-rs-php 默认参数"，不反映真实状态 | D.2-d 可从数据库读取参数来源标记和优化时间 |
| **普通解释文案** | 英文，偏技术但易懂 | 中文，当前解释已经比较好 | 保持现有中文风格，补充 retention 影响说明即可 |
| **重置卡片入口** | 不在此页面（在浏览器/卡片管理） | 显示"后续开放"但实际上管理页已有，说明过时 | 更新文案，提供跳转链接到 /review-cards/manage |

## 3. 产品原则

这些原则决定了 D 系列"做什么"和"怎么做"：

1. **用户不需要懂稳定度 / 难度公式。** 设置页的目标是让用户做决策，不是让用户学算法。所有说明和提示用"后果"导向的语言（"这样做会多复习 / 少复习"），不用公式参数导向的语言（"改变 w9 影响难度插值"）。

2. **不允许手动编辑卡片熟练度参数。** stability、difficulty 等数值由算法自动维护，设置页永远不会提供"手动调 stability"的功能。这是 FSRS 设计的核心前提。

3. **高级功能分阶段开放。** 第一版做信息展示和简单预估，后续版本逐步加入参数优化、模拟器、重排。避免一下子给用户太多复杂选项。

4. **所有会大幅改变到期时间的操作必须强提醒。** 参数优化、卡片重排这类操作可能产生几十张突然到期的卡，必须向用户说明影响量级，并要求确认。

5. **第一目标是"让用户知道自己改 retention 会带来什么后果"。** 当前用户改完 retention 后看不到任何影响，直到下次复习才体会出来。"帮我决定"不只是帮你决定，更是帮你理解决定的影响。

## 4. 小步骤拆解

以下任务按优先级排序，编号延续 [linguacafe-fsrs-roadmap.md](./linguacafe-fsrs-roadmap.md) 的 D 系列。

### D.1-scout（✅ 本任务已完成）

FSRS 设置页 Anki 对标侦察 + 独立拆解文档建立。完成本文件。

---

### D.2-a：FSRS 设置页信息架构重排

**目标**：把设置页从"功能说明 + 占位文字"变成真正的"控制面板"布局。

**要做什么**：
- 重新组织卡片布局，分成三个清晰区域：
  1. **复习目标**（Desired Retention + 帮助说明）
  2. **当前状态**（FSRS 统计总览，保持不动）
  3. **高级工具**（参数来源 / 优化 / 重排，占位或灰显）
- 更新过时文案：
  - "卡片重置 — 后续开放" → 改为"卡片重置 → 前往管理页"并加跳转链接
  - "参数优化 — 后续开放" → 保留，但放到高级工具区
  - 页面底部"FSRS 参数优化和卡片重置功能会在后续版本加入" → 删除或改为更准确的版本计划说明
- 不改算法，不新增后端接口。
- 不新增路由、不修改 API。

---

### D.2-b：Desired Retention 帮助说明增强

**目标**：帮用户理解不同 retention 值的实际影响。

**要做什么**：
- 在 retention 下拉框旁边或下方增加动态说明区域。
- 根据当前选择显示对应的影响描述（纯前端计算）：
  - 70% → "较低：复习次数少，但忘记概率较高"
  - 80% → "偏低：平衡偏向减少负担"
  - 85% → "中等偏低"
  - 90% → "默认平衡值，推荐大多数用户使用"
  - 92% → "偏高：记忆更牢固，但复习次数明显增多"
  - 95% → "很高：复习负担显著增加"
  - 97% → "极高：接近每日复习，仅适合少数用户"
- 不做模拟器，不做精确计算。纯文案引导。
- 不改后端。

---

### D.2-c：复习负担预估第一版

**目标**：给用户一个粗略的"改了 retention 后每天大概要多复习多少"的概念。

**要做什么**：
- 用当前 FSRS 统计总览已有数据（总词义卡数量、当前 retention、今日已复习），做一个非常粗略的预估。
- 预估逻辑（纯前端，不依赖后端计算）：
  - 基于当前 retention 对应的 FSRS 理论复习频率，估算如果调到目标 retention 后的日复习量。
  - 例如："当前 90%，今日复习了 25 张。调到 95% 后，预计每日复习 30-40 张。"
- 只做提示，不改变排程，不做参数优化。
- 第一版可以只在 Help Me Decide 区做文字说明，不画图表。

---

### D.2-d：参数来源与版本说明

**目标**：把"fsrs-rs-php 默认参数"这个固定文案变成真实可读的信息。

**要做什么**：
- 后端：在 Settings 表增加可选字段 `fsrs_parameters_source`（值：`default` / `optimized`）和 `fsrs_parameters_optimized_at`（ISO 8601 string）。
- 前端只读展示：
  - 如果 source = default：显示"当前使用 fsrs-rs-php 默认参数，尚未个性化优化"
  - 如果 source = optimized：显示"当前使用个性化参数（优化于 2026-06-XX）"
- 第一版只需一个后端 endpoint 返回这两个字段。不涉及参数编辑或优化执行。
- 不改参数结构、不改 FSRS 调度逻辑。

---

### D.3-scout：参数优化可行性侦察

**目标**：研究 fsrs-rs-php 是否支持 optimize，需要多少 ReviewLog。

**要做什么**：
- 只读研究：`vendor/open-spaced-repetition/fsrs-rs-php` 源码中是否有 `computeParameters` 或 `optimize` 方法。
- 评估需要的输入：多少条 ReviewLog、什么格式。
- 评估优化性能：用户有几百到几千条日志时是否会超时。
- 评估数据充分性：多少条日志以上才建议优化。
- 不改代码，不实现功能。只出侦察报告。

---

### D.3-a：参数优化第一版

**前置条件**：D.2-d（参数来源字段）+ D.3-scout（可行性确认）。

**目标**：让用户可以根据自己的复习历史生成个性化 FSRS 参数。

**要做什么**：
- 数据充分性检查：ReviewLog 数量 < 阈值（侦察后确定，建议 200+）时，显示警告"复习数据不足，不建议优化。"，按钮灰显。
- 优化触发：按钮"开始优化"，后端调用 fsrs-rs-php 的 optimize 方法。
- 优化结果预览：显示新参数与旧参数的对比表（w0-w19 的值变化不大的一般不显示详细）。
- 优化后保存：用户确认后保存到 Settings 表（建议存为 JSON，key 为 `fsrs_parameters`）。
- 不自动重排已有卡片。
- 不改变当前正在排队的卡片到期时间。

---

### D.4-scout：重排已有卡片可行性侦察

**目标**：评估改 retention 或参数后重排卡片的影响和风险。

**要做什么**：
- 只读研究：fsrs-rs-php 的 `reschedule` 方法。
- 评估影响量级：以当前用户数据模拟，看重排会导致多少卡片突然到期。
- 评估后端实现复杂度：需要逐卡调用 `fsrs.rs.repeat()` 还是可以批量？
- 不改代码。

---

### D.4-a：重排已有卡片手动工具

**前置条件**：D.4-scout 可行性确认。

**目标**：让用户可以在改 retention 或优化参数后手动选择重排卡片。

**要做什么**：
- 触发按钮放在高级工具区，明确标注"⚠️ 高级操作"。
- 强确认弹窗：显示"重排将影响 N 张启用中卡片，可能产生 M 张今日到期卡。此操作不可撤销。确认重排？"
- 后端逐卡调用 reschedule，更新 fsrs_due_at / fsrs_stability / fsrs_difficulty。
- 保留 ReviewLog（重排不产生日志，但也不删除现有日志）。
- 必须保留现有 `user_id` / `language_id` / `target_type=sense` / `confirmed` 安全约束。

## 5. 不做范围

以下功能明确不在 D 系列范围，除非后续单独侦察：

- ❌ 手动编辑单卡 stability / difficulty。
- ❌ 删除 ReviewLog。
- ❌ 绕过 sense-only（不做 word card 相关任何操作）。
- ❌ 修改 ReviewCard.target_type 语义。
- ❌ 修改 CSV / Anki TSV / JSON 导出。
- ❌ 修改复习评分按钮语义（again/hard/good/easy）。
- ❌ AnkiConnect 同步。
- ❌ 多 deck / preset 系统（除非后续单独侦察）。
- ❌ 移动端原生设置页。
- ❌ 图表化复习负担趋势图。
- ❌ 用户间参数共享 / 参数模板。

---

## 6. 推荐下一步

**推荐：D.2-a — FSRS 设置页信息架构重排。**

**理由**：
- 这是最小、最稳、最符合产品体验的第一步。
- 不碰算法、不新增后端、只改 Vue 布局和文案。
- 把设置页从"将来会做"的占位状态升级为"这页能帮我干什么"的控制面板。
- 为后续 D.2-b / D.2-c / D.2-d 奠定清晰的产品结构。
- 同时修复"卡片重置 → 后续开放"这个已经不准确的过时文案。

**实施预估**：单文件纯前端改动（`AdminReviewSettings.vue`），不含后端/路由/测试变更。
