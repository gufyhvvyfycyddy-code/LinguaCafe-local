## 2. Anki 的核心界面原则

### 原则 1：学习前有 Overview
用户打开 Anki 后看到的是牌组总览页，显示每个牌组的到期 / 新卡 / 学习中数量，以及一个"学习"按钮。用户不需要思考"今天先做什么"。

### 原则 2：复习页只服务回忆
复习页面的核心交互是：看问题 → 思考 → 点击显示答案 → 评分。所有 UI 元素都围绕这个流程。不把管理操作和复习操作混在一起。

### 原则 3：评分按钮承担调度反馈
Anki 的 Again/Hard/Good/Easy 按钮不仅评分，还显示"下次间隔"。用户评分前就能看到选择每个按钮的实际后果（1 分钟 / 10 分钟 / 明天 / 4 天）。信息透明，减少猜测。

### 原则 4：低频操作收进 More
Anki 把"编辑"、"标注"、"标记"、"暂停"、"置底"等操作放在一个"Edit"或"More"下拉菜单中。主复习区只保留评分和显示答案。

### 原则 5：Browser 是管理中心
Anki 的 Browser（卡片管理）是独立页面，有侧栏筛选、搜索框、可配置列、保存搜索、批量操作。它与复习页完全分离，互不干扰。

### 原则 6：Deck Options 是高级设置
Anki 把牌组设置放在单独的 Options 窗口中，作为"设置 → 牌组 → 选项"层级。用户在日常复习中完全看不到这些设置。

### 原则 7：搜索/筛选/保存搜索支持长期管理
Anki Browser 的核心竞争力是灵活筛选。用户可以做"已标记 + 到期 + 包含 X"这种组合筛选，并保存为常用搜索。

## 4. 逐页差距评估

### 4.1 首页 `/` → 对标 Anki Overview

| 维度 | 现状 | 差距 |
|------|------|------|
| 今日任务 | 隐式——用户需点入子页面 | 缺显式的"今日复习入口" |
| 复习量总览 | 无到期/新卡/学习中统计 | 需用户主动点入 SenseReview |
| 学习入口 | 无"开始学习"按钮 | 用户需知道要去 SenseReview |
| 日历 | 实时更新 | ✅ 已存在 |
| 统计 | 已实现 | ✅ 但未突出"今天应做什么" |

**影响**：用户第一次进入 LinguaCafe 后可能不清楚今天应该做什么。LinguaCafe 相比 Anki 缺了一个"今天你还有 XX 张卡需要复习，点击开始"的入口。

**风险等级**：中

**建议**：P1 — 在首页顶部增加"今日复习"卡片，显示到期/新/学习的数量，直接跳转到 SenseReview。

### 4.2 Sense 复习页 `/reviews/senses` → 对标 Anki Review Screen

| 维度 | 现状 | 差距 |
|------|------|------|
| 复习流程 | 问题 + 答案都可见 | ❌ 没有"先看问题→思考→显示答案"的节奏 |
| 评分按钮 | Again/Hard/Good/Easy | ✅ 已对齐 4 按钮 |
| 按钮显示间隔 | 不显示下次间隔 | ❌ 用户不知道选每个按钮的结果 |
| 低频操作 | 编辑/查看原文/归档/重置/删除全部在表面 | ❌ Anki 将这些收进 More |
| FSRS 数值 | 全部展开（到期/稳定度/难度/遗忘次数） | ❌ 90% 用户无需每天看 |
| 键盘快捷键 | 无 | ❌ 用户必须用鼠标点击评分 |
| 顶部 Stats | 8 个 chips | ❌ 信息密度过高 |

**影响**：SenseReview 是**最高频学习入口**，但当前设计偏向调试/管理风格，不是"最低认知负载的复习节奏"。用户需要处理的信息太多，学习体验不够流畅。

**风险等级**：高

**建议**：P0 — 重构 SenseReview 为核心复习页面：
- 先显示问题（lemma + 释义），隐藏答案，用户点击"显示答案"后才展示答案和评分按钮
- 每个评分按钮显示下次间隔（例如"1 分钟 / 10 分钟 / 明天 / 4 天"）
- 编辑/归档/重置/删除收进 More 下拉
- FSRS 数值默认折叠
- 支持键盘快捷键（1/2/3/4 或空格）

### 4.3 词义确认页 `/senses/review` → 对标 Anki Browser (但 LinguaCafe 独有)

| 维度 | 现状 | 差距 |
|------|------|------|
| 审核流程 | GPT 推荐 → 用户确认 | ✅ 流程独特且完整 |
| 批量操作 | 有 | ✅ |
| JSON/evidence | 展开（疑似） | ❌ 应默认折叠 |
| 用户决策 | 确认/改绑/新建/忽略 | ✅ 但文案偏技术 |
| 产品定位 | 调试后台感 | ❌ 应更像"学习材料整理台" |

**影响**：这是 LinguaCafe 独有的 AI 词义审核台，Anki 没有直接对标。当前设计偏技术调试后台。

**风险等级**：中低

**建议**：P3 — 在刷卡页优化之后，将此页面产品化。折叠 JSON evidence、突出"这个词在这句话里的意思"、简化用户决策路径。

### 4.4 复习卡管理页 `/review-cards/manage` → 对标 Anki Browser

| 维度 | 现状 | 差距 |
|------|------|------|
| 搜索 | 文本搜索 lemma/释义/例句 | ✅ 已有 |
| 筛选 | 9 个按钮（全部/到期/未来/归档等） | ✅ 接近 |
| 侧栏 | 无 | ❌ Anki 有侧栏做快速筛选 |
| 保存搜索 | 无 | ❌ Anki 可保存搜索 |
| 列配置 | 固定列 | ❌ Anki 可自定义列 |
| 批量操作 | 有 | ✅ 但需要收口 |
| 详情预览 | 无 | ❌ Anki 有选中卡片预览区 |

**影响**：管理页已经很强，接近 Anki Browser。主要差距在侧栏、保存搜索、列配置。

**风险等级**：低

**建议**：P2 — 从"继续加列"转向"组织结构优化"：加侧栏筛选项、保存搜索功能。不要继续无限增加右侧列。

### 4.5 Legacy 复习 `/review` → 对标

**问题**：这句路由仍存在，承担旧的 word/phrase/sense 混合逻辑。与 sense-only 主线冲突。

**风险等级**：中

**建议**：降级标记为 legacy，或统一跳转到 `/reviews/senses`。不要在 roadmap 上继续维护。

### 4.6 阅读页 + 点词面板 `/chapters/read/:id`

| 维度 | 现状 | 差距 |
|------|------|------|
| 阅读体验 | Vuetify reader | ✅ 可用 |
| 点词 | 弹出词义面板 | ✅ 但状态反馈不足 |
| 词状态 | 未清晰区分"新词/学习中/已掌握" | ❌ 颜色系统存在但不够直观 |
| 词义闭环 | 点词确认 → 进入复习 | ✅ 但链路较长 |

**影响**：点词面板是复习闭环的前端入口之一。如果用户点了词但不知道会触发什么（是否进复习、是否创建 ReviewCard），容易造成困惑。

**风险等级**：中低

**建议**：P4 — 阅读页状态标识优化：点词面板清晰显示"此词义已确认 / 待确认 / 学习中 / 已掌握"，以及点确认后会发生什么。

### 4.7 FSRS 设置页 `/admin/reviews`

| 维度 | 现状 | 差距 |
|------|------|------|
| D.4 | 已收口 | ✅ 完成 |
| 重排 preview | 有 | ✅ |
| 重排确认 | 有 | ✅ |
| 撤销 | 有 | ✅ |
| 复杂度 | 中高 | ❌ 应折叠成高级工具 |

**影响**：设置页已完成算法安全主线。现在功能太多，应有更好的层级组织。

**风险等级**：低

**建议**：P4 — D.4 收口后，把重排/撤销折叠成"高级工具"区域，不作为默认可见区。

## 5. 当前 roadmap 评估

### 5.1 已完成的主线
- ✅ FSRS 算法集成（D.1-D.3）
- ✅ Sense-only 复习系统（主线）
- ✅ 重排预览 + 确认 + 撤销（D.4）
- ✅ 前端按钮、弹窗、倒计时、安全边界

### 5.2 当前 roadmap 偏向
当前 roadmap（`linguacafe-fsrs-roadmap.md`）几乎完全聚焦于：
- FSRS 参数优化
- 重排调度安全
- 管理员设置页

### 5.3 评估结论
**当前 roadmap 过度偏向 FSRS 设置页和管理页。**

原因：
1. 最近 20+ 个子任务全部围绕 D.3/D.4（设置页和重排安全）。
2. 用户每天实际使用的页面（刷卡页、首页、阅读页）长期未优化。
3. Anki 对标提示最大的体验差距在复习页，不在设置页。
4. 当前 roadmap 缺少"学习体验"主线。

### 5.4 调整建议
| 旧计划 | 建议 | 原因 |
|--------|------|------|
| D.5（未知功能） | ⏸ 暂缓 | 应先优化已有体验 |
| FSRS 设置页继续折叠 | ⏸ 延后至 P4 | 设置页已安全，不急需 |
| 刷卡页学习节奏重构 | 🆕 加入 P0 | 最高频学习入口 |
| 首页 Overview 入口 | 🆕 加入 P1 | 用户首次进入体验 |
| 管理页 Browser 化 | 🆕 加入 P2 | 已有基础，加侧栏和搜索保存 |
| 词义确认页产品化 | 🆕 加入 P3 | 独有功能但偏后台 |
| Playwright 登录修复 | 🆕 加入 P5 | 辅助验收能力提升 |

## 7. 建议下一步任务

### 推荐：`UI-Anki-Review-Scout` — 刷卡页学习节奏重构侦察

**不是直接改代码**，而是先做一次集中侦察：

1. 阅读 Anki 刷卡的界面原则（问题→答案→评分→间隔显示）。
2. 详细审查 SenseReview.vue 的：
   - 信息分区是否合理
   - 高频操作、中频操作、低频操作是否应分层
   - 评分按钮是否应显示间隔
   - 键盘快捷键
   - 复习节奏是"先看问题"还是"全展示"。
3. 输出一份对比报告和分步骤修改方案。

- 刷卡页是用户的最高频交互界面
- 当前 SenseReview 的认知负载远高于 Anki 同行
- 优化此页面带来最直接的用户体验提升
- 侦察结果决定了后续 UI 改造的具体方向和步骤

### 不推荐直接进入

- ❌ FSRS 设置页折叠（优先级低）
